Project Description ¶

Receiver for the ACARS protocol

The ACARS protocol, widely used in plane-to-ground communications, is old enough to be well documented and easily decoded using Software Defined Radio (SDR). I have used this protocol, since I am not aware of any other opensource ACARS decoding project still active, as an opportunity to learn programming in the GNURadio envionment. A detailed description of my working methodology, published in the October 2012 French issue of GNU/Linux Magazine  http://jmfriedt.free.fr/lm_sdr.pdf, is available as an English translation of the original article at  http://jmfriedt.free.fr/en_sdr.pdf. Someone,  http://foxgulch.com/WordPress/?p=640 managed to get this example running so it seems functional enough for anyone to test.

Currently, the decoder is very verbose: upon detecting a transmission, it will start by printing a few information such a threshold value (which I believe to be the parameter limiting the receiver performance at the moment: should be dynamically tuned rather than using the fixed value of 150 experimetally defined at the moment), and then start decoding. Identifying a transmission and decoding its content are two very separate steps: the former seems to be working very well, the second less, maybe because I want to decode all the received messages rather than selecting only the ones with the best signal to noise ratio.

The decoder displays all the raw decoded byte values: if the sentence does not start with the magic sequence 2B 2A 16 16 01, then chances are that the sentence will not be decoded correctly. Yet, I am hoping that maybe some day we will find a way to correct a missing bit (which then propagates to make the whole message unreadable) and thus the raw byte values might be useful. I then interpret each printable character as an ASCII value and print the resulting message. If the sentence makes sense, I finally interpret the content in terms of plane reference, flight reference, start of text (STX), message content, and end of text (ETX). Each recorded sentence is tagged with the computer date at the time of the reception.


How To Install ¶

The decoder requires the libfftw3 library.

(from trunk) cd build
cmake .. && make && sudo make install


How To Run ¶

A .grc file for setting the receiver frequency (131.725 MHz for EU, 136.800 MHz for NAm), demodulating the AM signal and sending the result to the decoder is provided as acars_test.grc, for use with gnuradio-companion. Just execute the grc from gnuradio-companion and enjoy the decoded sentences displayed either in the terminal emulator at the bottom of gnuradio-companion, or in the xterm from which gnuradio-companion was lauched. A log file is created in /tmp which stores all displayed data for further processing and storage.

Currently, the decoder has only been tested with an EZCAP DVB tuner using the OsmoSDR receiver block. Reports of tests with the UHD are most welcome.

A few examples of gr-acars running at the same time than listening to voice communication can be downloaded at  http://jmfriedt.free.fr/acars_audio1.mp4 or  http://jmfriedt.free.fr/acars_audio2.mp4 or  http://jmfriedt.free.fr/acars_audio3.mp4 or  http://jmfriedt.free.fr/acars_audio4.mp4 or  http://jmfriedt.free.fr/acars_audio5.mp4 or  http://jmfriedt.free.fr/acars_audio6.mp4 or  http://jmfriedt.free.fr/acars_audio7.mp4. In this example, canals are not filtered so that voice communication prevents efficient ACARS decoding, hence the high failure rate. Nevertheless, ACARS sentences are well decoded when the 1 MB wide band is not busy with other activity.


Known bug ¶

Only one occurence of the decoder can appear in a grc graph, preventing decoding multiple ACARS channels simultaneously. The cause has been tracked to the (lack of) FFTW thread safety issue during the plan allocation step.
